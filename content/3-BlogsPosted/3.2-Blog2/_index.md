# Building a File Upload Service on AWS with Presigned URLs

## 1. Research Overview

While studying application architecture on AWS, a common requirement is building a file upload feature for web or mobile applications. A simple initial approach is to let the client upload files to the backend, and then have the backend store those files in Amazon S3.

However, when user traffic grows or file sizes become larger, this approach forces the backend to process large amounts of data unnecessarily. The backend must receive the file and then transfer it to S3, which can increase system load, latency, and operating cost.

The AWS recommended approach is to use **Presigned URLs** together with serverless services such as Amazon API Gateway, AWS Lambda, and Amazon S3. In this model, the backend authenticates the user and generates a temporary upload URL, while the client uploads the file directly to S3.

![File upload architecture with S3 Presigned URL](/images/blog2-s3-presigned-url.jpg)

## 2. Limitations of Backend-Based Uploads

If every file passes through the backend, the system may face several limitations:

* The backend processes large data payloads even though it does not need the file content directly.
* API response time can increase when files are large or many requests happen concurrently.
* Backend capacity must be scaled mainly to support file transfer.
* Error handling, timeout handling, and retry logic become more complex.
* Compute and bandwidth costs may increase unnecessarily.

For this reason, the backend should not always act as the data transfer layer for file uploads. Instead, it should focus on business control, authentication, and permission management.

## 3. Presigned URL Solution

A Presigned URL is a temporary URL generated by AWS that allows a client to perform a specific operation on Amazon S3 within a limited time. It can be used to upload or download an object without giving direct AWS credentials to the end user.

The basic upload workflow is:

| Step | Component | Responsibility |
| --- | --- | --- |
| 1 | Client | Requests an upload URL |
| 2 | Amazon API Gateway | Receives the client request |
| 3 | AWS Lambda | Authenticates the user and generates a Presigned URL |
| 4 | Amazon S3 | Receives the direct file upload from the client |
| 5 | S3 Event or EventBridge | Emits an event when the object is created |
| 6 | Lambda Processor | Processes the file after upload |
| 7 | DynamoDB, SNS, CloudWatch | Stores metadata, sends notifications, and records monitoring logs |

The key point is that AWS Lambda does not process the file data directly. Lambda only generates the Presigned URL based on IAM permissions, and the client uses that URL to upload directly to S3.

## 4. Role of AWS Services

### 4.1. Amazon API Gateway

Amazon API Gateway acts as the system entry point. The client sends a request to API Gateway to obtain an upload URL. API Gateway helps manage endpoints, authentication, authorization, request throttling, and integration with Lambda.

### 4.2. AWS Lambda

AWS Lambda handles business logic such as validating the user, checking file type, checking file size limits, and generating the Presigned URL. Because Lambda does not receive the file directly, processing time is shorter and compute cost is reduced.

### 4.3. Amazon S3

Amazon S3 stores the uploaded files. S3 provides high scalability, high durability, and is suitable for many file types such as images, documents, videos, and application data.

### 4.4. AWS IAM

AWS IAM controls permissions between services. Lambda must have the appropriate permissions to generate Presigned URLs for the S3 bucket. IAM permissions should follow the least privilege principle and only allow the required actions on the relevant bucket or prefix.

### 4.5. Amazon EventBridge and S3 Event Notification

After a file is uploaded successfully, S3 can emit an event. This event can trigger another Lambda function or be routed through EventBridge for post-upload processing.

## 5. Post-Upload Processing

Uploading the file to S3 is not always the final step. After an object is created, the system can continue with asynchronous tasks such as:

* Resizing images.
* Scanning files for viruses or unsafe content.
* Storing metadata in DynamoDB.
* Sending notifications through Amazon SNS.
* Recording logs and metrics in Amazon CloudWatch.
* Creating thumbnails or optimized display versions.

Separating upload from post-upload processing makes the system easier to scale. Users do not need to wait for all processing tasks to finish, while backend services can continue handling additional work asynchronously.

## 6. Benefits of the Serverless Architecture

This architecture is a good example of assigning a clear responsibility to each AWS service:

* API Gateway receives and manages requests.
* Lambda handles business logic and generates upload URLs.
* S3 stores files directly from clients.
* EventBridge or S3 Event Notification triggers post-upload workflows.
* DynamoDB stores metadata.
* SNS sends notifications.
* CloudWatch supports logging and monitoring.

With this separation, the system becomes simpler and can scale naturally with demand without requiring server management.

## 7. Deployment Notes

When implementing file uploads with Presigned URLs, several points should be considered:

* Set an appropriate URL expiration time to reduce misuse risk.
* Restrict allowed file types and file sizes.
* Use least-privilege IAM policies for Lambda.
* Configure CORS for the S3 bucket when the client is a web application.
* Avoid allowing clients to freely choose object keys if overwrite risk exists.
* Enable logging and monitoring with CloudWatch to track upload errors.
* Add post-upload security checks when files come from external users.

## 8. Conclusion

Using Presigned URLs helps build a more efficient file upload feature on AWS. The backend no longer needs to process file data directly, reducing system load and latency while taking advantage of Amazon S3 scalability.

For web or mobile applications, combining API Gateway, Lambda, S3, EventBridge, and post-upload processing services provides a flexible, scalable, and cost-efficient serverless architecture. This is a practical model to consider when designing file upload systems on AWS.
