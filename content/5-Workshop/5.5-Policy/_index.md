# Application Deployment and Delivery

## React, S3, and CloudFront

```bash
cd frontend
npm install
npm run build
aws s3 sync ./dist s3://<frontend-bucket>
aws s3 cp ./dist/index.html s3://<frontend-bucket>/index.html \
  --content-type "text/html" \
  --cache-control "no-cache, no-store, must-revalidate"
```

After synchronization, the bucket root must contain `index.html`, the `assets/` directory, and the required icons. `index.html` is the SPA entry point and becomes the CloudFront Default root object.

![Frontend S3 bucket root with index.html and assets](/5-workshop/document-security/img-22-frontend-bucket-root.png)

Do not use `--delete` before verifying the target scope. Configure CloudFront with `index.html` as the default root object, a private S3 OAC default behavior, a cache-disabled `/api/*` behavior to EC2/Nginx, HTTP-to-HTTPS redirect, and SPA 403/404 fallback to `/index.html` with HTTP 200.

The `assets/` directory contains the JavaScript and CSS bundles generated by Vite. Hashed file names allow browsers and CloudFront to cache each deployment safely.

![JavaScript and CSS bundles in the frontend S3 assets directory](/5-workshop/document-security/img-22-frontend-assets.png)

The distribution uses two origins: Amazon S3 for the frontend and `SecurityPortalBackend` for the API hosted on EC2.

![CloudFront origins for the S3 frontend and EC2 backend](/5-workshop/document-security/img-23-cloudfront-origins.png)

The `/api/*` behavior routes requests to the backend with caching disabled and the required viewer information forwarded. The default behavior serves the S3 frontend with optimized caching. Both redirect HTTP to HTTPS.

![CloudFront behaviors for the backend API and S3 frontend](/5-workshop/document-security/img-23-cloudfront-behaviors.png)

```bash
aws cloudfront create-invalidation \
  --distribution-id <distribution-id> \
  --paths "/*"
```

After the distribution finishes deploying, open the CloudFront domain over HTTPS and test the `/login` route. The complete UI, icons, bundles, and sign-in form must load without exposing the S3 bucket directly.

![Document Security website delivered through CloudFront](/5-workshop/document-security/img-24-cloudfront-website.png)

## EC2, Nginx, and Gunicorn

```text
CloudFront /api/* -> Nginx :80 -> Gunicorn/Flask :5000
```

Before deploying the backend, confirm that the EC2 instance is **Running** and all status checks have passed. This lab uses a `t3.micro` instance in Availability Zone `ap-southeast-1a`.

![Backend EC2 instance Running with all status checks passed](/5-workshop/document-security/img-25-ec2-running.png)

```bash
cd /app/backend
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

Run Gunicorn through `systemd` with `Restart=always`, an explicit working directory, and a protected environment file. Use this Nginx location:

```nginx
location /api/ {
    proxy_pass http://127.0.0.1:5000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    client_max_body_size 20M;
}
```

## Health checks

```bash
curl -i http://<backend-origin>/api/health
curl -i https://<cloudfront-domain>/api/health
```

The response should show the backend, S3, and all three DynamoDB tables as `ok: true`. Any failed dependency should make the overall state `degraded`.

The following response confirms that `/api/health` works through CloudFront. The backend, S3, Documents, SecurityIncidents, and DocumentVersionAudit checks all return `ok: true`.

![API health-check response through CloudFront](/5-workshop/document-security/img-26-cloudfront-health-check.png)

{{% notice info %}}
For production, place the backend behind an Application Load Balancer, use HTTPS to the origin, and consider an Auto Scaling Group.
